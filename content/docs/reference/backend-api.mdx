title: Backend API

The Backend API is the administrative control plane for your Wacht deployment. It is a server-to-server API used to provision and manage users, configure authentication settings, administer OAuth applications, inspect sessions, manage webhooks, and access analytics. It is never called from the browser or exposed to end users.

## Base URL

```
https://api.wacht.dev
```

All requests go to this shared endpoint. The deployment you are managing is identified by the API key used to authenticate.

## Authentication

Every request to the Backend API must include an `Authorization` header with a valid API key:

```bash
curl https://api.wacht.dev/users \
  --header "Authorization: Bearer $WACHT_API_KEY"
```

### Obtaining API keys

API keys are managed in the Wacht console under **Settings ã API Keys**. Each key is scoped to a single deployment and can be granted limited permissions.

- **Secret keys** (`wacht_live_sk_...`) have full access. Store them in environment variables and never commit them to source control.
- **Restricted keys** can be scoped to specific resource types and operations for least-privilege access.

```bash
# Recommended: load from environment
curl https://api.wacht.dev/users \
  --header "Authorization: Bearer $WACHT_API_KEY"
```

### Key security

Backend API keys carry full administrative access to your deployment. If a key is compromised, rotate it immediately from the console. Keys cannot be recovered after initial creation ã store them securely on first display.

## Making requests

Most request bodies are JSON with `Content-Type: application/json`. Endpoints that accept files, such as user profile images, use `multipart/form-data`.

```bash
curl https://api.wacht.dev/users \
  --request POST \
  --header "Authorization: Bearer $WACHT_API_KEY" \
  --form "first_name=Jane" \
  --form "last_name=Doe" \
  --form "email_address=jane@example.com"
```

## Response format

### Success

Successful responses return `2xx` with a JSON body wrapping the result in `data`:

```json
{
  "data": {
    "id": "usr_01j...",
    "first_name": "Jane",
    "last_name": "Doe",
    "primary_email_address": "jane@example.com",
    "profile_picture_url": "",
    "created_at": "2025-01-15T10:30:00Z"
  }
}
```

### Paginated lists

List endpoints return offset-based pagination. Use `limit` and `offset` query parameters to page through results:

```bash
curl "https://api.wacht.dev/users?limit=50&offset=0" \
  --header "Authorization: Bearer $WACHT_API_KEY"
```

```json
{
  "data": [ ... ],
  "total": 124,
  "limit": 50,
  "offset": 0
}
```

Increase `offset` by `limit` to fetch the next page.

## Error handling

The API uses standard HTTP status codes. All error responses include a `code` for programmatic handling and a `message` for debugging:

```json
{
  "error": {
    "code": "user_not_found",
    "message": "No user exists with the provided ID.",
    "status": 404
  }
}
```

| Status | Meaning |
|--------|---------|
| `400` | Bad request ã malformed JSON or missing required fields |
| `401` | Unauthorized ã API key missing, invalid, or revoked |
| `403` | Forbidden ã key lacks permission for this operation |
| `404` | Not found ã resource does not exist in this deployment |
| `409` | Conflict ã e.g. user with that email already exists |
| `422` | Unprocessable ã request was well-formed but failed validation |
| `429` | Rate limited ã reduce request frequency |
| `500` | Server error ã contact support if this persists |

Always check `error.code` rather than `error.message` ã messages are for humans and may change.

## Rate limiting

Requests are rate-limited per API key. The limit is applied over a rolling window. When exceeded, the API returns `429 Too Many Requests` with a `Retry-After` header:

```
HTTP/1.1 429 Too Many Requests
Retry-After: 2
```

Implement exponential backoff when building automation pipelines that issue many requests in sequence.

## Endpoints

### Users ã 16 operations
`Users` ôñ `Invitations`

Create, retrieve, update, and delete user accounts. Search and filter users by email, name, or metadata. Manage pending invitations ã create, list, revoke, and resend them.

### Authentication ã 42 operations
`API Auth` ôñ `Session` ôñ `OAuth` ôñ `JWT Templates` ôñ `Signing Keys` ôñ `Deployment Credentials` ôñ `Session Tickets`

Create and manage API keys with configurable scopes. List and revoke active sessions across users. Administer OAuth applications and their clients ã create apps, manage secrets, and configure redirect URIs. Rotate OIDC signing keys per OAuth app, or mark them compromised on suspected leak. Configure JWT template overrides for custom claims. Mint fresh deployment credentials (publishable key, hosts, and a one-shot API key) for bootstrap pipelines. Issue server-side session tickets (impersonation, agent access, webhook-app access, API-auth access) that hand the user a short-lived redeemable URL.

### Organizations & Workspaces ã 29 operations
`Organizations` ôñ `Workspaces` ôñ `Invitations`

Create and manage organizations and their settings. Control membership, roles, and permissions. Create and configure workspaces within organizations. Manage workspace members and transfer ownership. Issue, list, and discard organization invitations ã optionally scoped to a workspace and role.

### Platform ã 128 operations
`AI` ôñ `Analytics` ôñ `Notifications` ôñ `Webhooks` ôñ `Settings` ôñ `Segments`

Administer AI actors, projects, and threads. Access deployment analytics and usage metrics. Manage notification templates and delivery. Configure webhook endpoints, inspect delivery logs, and replay failed events. Manage global deployment settings. Define user and organization segments for targeting.

### Other ã 3 operations
`Waitlist` ôñ `Default`

Manage waitlist entries and enrollment status. Miscellaneous utility endpoints.
